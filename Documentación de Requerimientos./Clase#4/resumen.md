# Documentación de requerimientos.

## ¿Por qué documentar?

Documentar es importante porque evita que la información se pierda y permite que todo el equipo tenga acceso a los requerimientos del sistema.

Mientras la información solo está en la memoria puede olvidarse, cambiar o perderse cuando alguien deja el proyecto. En cambio, la documentación permanece y puede revisarse.

## Funciones de la documentación

La documentación cumple varias funciones importantes:

- **Acuerdo:** define lo que cliente y equipo acordaron construir.
- **Guía:** orienta a los desarrolladores sobre qué hacer.
- **Verificación:** ayuda a comprobar si el sistema cumple lo prometido.
- **Conocimiento:** conserva la información del proyecto.
- **Legal:** sirve como evidencia en auditorías o conflictos.

La cantidad de documentación depende del nivel de riesgo del proyecto.

## Estándar IEEE 830

El IEEE 830 es un estándar internacional para documentar requerimientos de software mediante un documento llamado **SRS (Software Requirements Specification)**.

Su objetivo es organizar los requerimientos de forma clara y comprensible para cualquier profesional.

### Beneficios

- Estructura clara
- Facilita la comunicación
- Ayuda en auditorías
- Reduce olvidos

### Limitaciones

- Puede ser muy extenso para proyectos pequeños
- Requiere tiempo y disciplina

## Características de un buen SRS

Un SRS debe ser:

- Correcto
- No ambiguo
- Completo
- Consistente
- Verificable
- Modificable
- Trazable

## Casos de uso

Un caso de uso describe paso a paso cómo un usuario interactúa con el sistema para lograr un objetivo.

Incluye:

- Actor
- Objetivo
- Flujo principal
- Flujos alternos
- Excepciones
- Reglas de negocio

Los casos de uso explican **cómo sucede una acción dentro del sistema**, no solo qué hace.

## Plantilla de un caso de uso

Un caso de uso profesional suele incluir:

- ID
- Nombre
- Actor principal
- Descripción
- Precondiciones
- Postcondiciones
- Flujo principal
- Excepciones
- Prioridad

## Estructura del SRS IEEE 830

El documento SRS se organiza en:

1. **Introducción**
2. **Descripción general**
3. **Requisitos específicos**
4. **Apéndices**

También incluye identificadores únicos como RF-001, RNF-001 o CU-001 para facilitar la trazabilidad.

## Revisión cruzada

La revisión cruzada consiste en que otras personas revisen el documento para detectar errores.

Se busca identificar:

- Ambigüedades
- Contradicciones
- Información faltante
- Requerimientos no medibles
- Dependencias no claras

## Conclusión

La documentación permite organizar y proteger la información de un proyecto. Herramientas como el estándar IEEE 830, los casos de uso y la revisión cruzada ayudan a crear sistemas más claros, verificables y de mejor calidad.
